4. HIGH LEVEL DESIGN (HLD)
				4.1 Purpose
***********************************************************************************************************
The High-Level Design (HLD) defines the overall architecture of the AI-Powered E-Commerce Backend Microservices Platform. It describes the system components, microservices, communication mechanisms, infrastructure, database architecture, security, deployment model, and technology stack before implementation.
The system follows a cloud-native microservices architecture with Domain-Driven Design (DDD), Event-Driven Architecture (EDA), Database per Service, and Polyglot Persistence.


				4.2 Architecture Goals
***********************************************************************************************************
The architecture is designed to provide:
 ● High Availability
 ● High Scalability
 ● Loose Coupling
 ● High Cohesion
 ● Independent Deployment
 ● Fault Isolation
 ● Event-Driven Communication
 ● Secure Authentication & Authorization
 ● Observability
 ● Cloud-Native Deployment
 ● Polyglot Persistence

				4.6 Microservices
***********************************************************************************************************
			Identity Service
Responsibilities
 ● User Registration
 ● Login
 ● JWT Generation
 ● User Profile
 ● Address Management
 ● Role Management

Database 
 ● MySQL

Reason 
 ● Identity data is highly relational and requires ACID transactions.

			Catalog Service
Responsibilities
 ● Product Management
 ● Category Management
 ● Brand Management
 ● Product Reviews
 ● Product Search
 ● Filtering
 ● Sorting

Database 
 ● MongoDB

Reason 
 ● Products have dynamic attributes.

Example:
Laptop : RAM,CPU,GPU
Phone : Battery,Camera,SIM
Shoes : Size,Color,Material
MongoDB handles flexible schemas efficiently.

			Inventory Service
Responsibilities
 ● Stock Management
 ● Reserve Stock
 ● Release Stock
 ● Inventory Validation

Database 
 ● PostgreSQL

Reason 
 ● Inventory requires strong consistency and transactional updates to prevent overselling.

			Cart Service
Responsibilities
 ● Add Item
 ● Remove Item
 ● Update Quantity
 ● View Cart
 ● Cart Expiry

Database 
 ● Redis

Reason 
 ● Shopping carts require extremely fast read/write performance and automatic expiration (TTL).

			Order Service
Responsibilities
 ● Checkout
 ● Place Order
 ● Order History
 ● Cancel Order
 ● Order Tracking

Database 
 ● PostgreSQL

Reason 
 ● Orders are transactional and require reliable ACID guarantees.

			Payment Service
Responsibilities
 ● Payment Processing
 ● Refund
 ● Payment History

Database 
 ● PostgreSQL

Reason 
 ● Financial transactions require strong consistency, durability, and transactional integrity.

			Notification Service
Responsibilities
 ● Email
 ● SMS
 ● Push Notifications
 ● Notification Preferences

Database 
 ● MongoDB

Reason 
 ● Notification payloads vary in structure and are naturally represented as documents.

			AI Service
Responsibilities
 ● Product Recommendation
 ● Semantic Search
 ● Chat Assistant
 ● Review Summary

Databases 
 ● MySQL
 ● Elasticsearch

Reason 
 ● MySQL stores operational data such as chat history, recommendation requests, and AI metadata.

Elasticsearch powers:
 ● Semantic Search
 ● Full-text Search
 ● Product Ranking
 ● Recommendation Indexes

				4.8 Database Architecture
***********************************************************************************************************
The platform follows the Database per Service pattern.

Service							Database
Identity						MySQL
Catalog							MongoDB
Inventory						PostgreSQL
Cart							Redis
Order							PostgreSQL
Payment							PostgreSQL
Notification						MongoDB
AI							MySQL + Elasticsearch

Each microservice owns its own persistence layer.
No microservice accesses another microservice's database directly.
All communication occurs through REST APIs or Kafka events.
